# 家承项目开发规则

> 四条核心原则，指导代码编写和系统设计

---

## 1. Think Before Coding（先思后码）

### 含义
在动手写代码之前，先花时间思考问题的本质、解决方案的合理性、以及可能的影响。

### 实践要点
- **理解需求**：明确这个功能要解决什么问题，用户是谁，使用场景是什么
- **方案对比**：列出至少 2-3 种实现方案，评估优缺点
- **影响分析**：考虑对现有代码、数据库结构、用户体验的影响
- **边界情况**：思考异常流程、错误处理、极限情况

### 示例
❌ 错误：接到"添加搜索功能"的需求，立即开始写代码
✅ 正确：先思考——搜索范围是什么？按姓名还是按全文？是否需要实时搜索？数据量大时怎么优化？

### 时间分配建议
- 简单功能：思考 5 分钟，编码 15 分钟
- 复杂功能：思考 30 分钟，编码 2 小时
- 架构调整：思考半天，编码一天

---

## 2. Simplicity First（简洁优先）

### 含义
选择最简单、最直接、最易于理解的实现方式。避免过度工程化。

### 实践要点
- **KISS 原则**：Keep It Simple, Stupid
- **避免 premature optimization**：先让代码工作，再考虑优化
- **可读性 > 巧妙性**：代码是写给人看的，顺便给机器执行
- **减少依赖**：每引入一个库，就增加一份维护成本

### 示例
❌ 错误：为了"可扩展性"，设计复杂的插件系统，而实际上只需要 2-3 种固定类型
✅ 正确：先用 `if/else` 或 `switch` 处理，等真正有 10+ 种类型时再抽象

### 判断标准
- 一个新同事能在 5 分钟内看懂这段代码吗？
- 删除这段代码会影响多少其他地方？
- 这个方案需要多少文档来解释？

---

## 3. Surgical Changes（精准修改）

### 含义
像外科手术一样精准地修改代码——只动该动的地方，最小化改动范围，避免"附带伤害"。

### 实践要点
- **最小改动原则**：只修改实现功能所必需的代码
- **避免重构诱惑**：修 bug 时不要顺手重构无关代码
- **原子化提交**：一个 commit 只做一件事
- **回滚安全**：确保改动可以单独回滚，不影响其他功能

### 示例
❌ 错误：修复登录页面的样式问题时，顺手重写了整个认证系统
✅ 正确：只修改登录页面的 CSS，认证系统的逻辑原封不动

### 工具支持
- 使用 `git diff` 检查改动范围
- 使用 `git add -p` 分块提交
- Code Review 时关注"为什么改这里"

---

## 4. Goal-Driven Execution（目标驱动）

### 含义
始终以最终目标为导向，不做与目标无关的事情。每个功能、每行代码都要服务于明确的目标。

### 实践要点
- **明确目标**：这个功能要解决什么具体问题？成功标准是什么？
- **拒绝镀金**：不做需求之外的功能
- **快速验证**：用最小成本验证方案是否可行
- **及时止损**：如果发现方向错误，立即调整或放弃

### 示例
❌ 错误：用户需要"导出 PDF"，结果实现了完整的文档编辑系统
✅ 正确：先用最简单的方式生成 PDF（如浏览器打印转 PDF），验证用户真的需要后再完善

### 目标检查清单
- [ ] 这个功能有明确的用户价值吗？
- [ ] 有明确的完成标准吗？
- [ ] 这是达成目标的最短路径吗？
- [ ] 如果砍掉这个功能，用户会受影响吗？

---

## 四条规则的关系

```
Think Before Coding
       ↓
   明确方向
       ↓
Simplicity First
       ↓
   选择路径
       ↓
Surgical Changes
       ↓
   精准执行
       ↓
Goal-Driven Execution
       ↓
   达成目标
```

1. **先思考** → 避免盲目编码
2. **选简单** → 降低复杂度
3. **精准改** → 减少风险
4. **向目标** → 确保价值

---

## 应用到当前项目

### 家承 SaaS 的践行方式

| 规则 | 家承项目实践 |
|------|-------------|
| Think Before Coding | 新增功能前先画流程图，明确数据流 |
| Simplicity First | 族谱可视化先用 CSS 实现，而非引入 D3.js |
| Surgical Changes | 修复登录问题时不动数据库结构 |
| Goal-Driven Execution | MVP 阶段不做 AI 智能推荐，先做好基础编辑 |

---

## 违反规则的代价

| 规则 | 违反后果 |
|------|---------|
| 不思考就编码 | 返工、重构、技术债 |
| 过度复杂 | 维护困难、新人上手慢、bug 多 |
| 大范围修改 | 引入新 bug、回滚困难、影响线上 |
| 偏离目标 | 做无用功、延期、用户不买账 |
